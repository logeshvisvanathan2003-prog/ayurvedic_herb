# Blockchain Layer — Ayurvedic Herb Traceability (Docker-free Hyperledger Fabric)

## Solution summary (for your report/PPT)

This system implements end-to-end botanical traceability for Ayurvedic herbs
on a permissioned Hyperledger Fabric blockchain, combined with geo-tagged
data capture and consumer-facing QR verification. Farmers and wild
collectors register a **CollectionEvent** with GPS coordinates, species,
harvest date and quantity; a chaincode-enforced smart contract checks this
against registered **conservation zones** — rejecting or flagging harvests
outside approved geo-fenced regions, outside the approved season, or beyond
a species' seasonal collection quota, directly enforcing National Medicinal
Plants Board and Good Agricultural and Collection Practices guidelines.
As the herb moves through **ProcessingStep** (drying, grinding, storage) and
**QualityTest** (moisture, pesticide residue, DNA authentication) stages,
every event is submitted as a signed, endorsed transaction to the Fabric
ledger, giving an immutable, cryptographically-verifiable audit trail no
single party can alter after the fact. When herbs are moved between
custodians — farmer to processor, processor to lab, lab to manufacturer —
a **single-use QR "custody token"** is minted for that specific leg; it can
be redeemed exactly once at delivery, and GPS/time-elapsed checks flag
physically implausible transit (the mechanism that directly targets
transport-stage fraud: cloned QR labels, phantom deliveries, and diverted
shipments are all caught automatically). At the point of sale, a unique
product QR code is generated on-chain and printed on packaging; scanning it
opens a lightweight web portal (no app install required) that queries the
ledger for the complete **Provenance** bundle — farm origin with an
interactive map, processing and lab certificates, the full transport chain
of custody, and a live blockchain-integrity badge — giving consumers,
retailers and regulators a single, tamper-evident source of truth from
farm to shelf, while also enabling instant, ledger-driven product recalls.

## Is this "real" Hyperledger Fabric? Yes — and no Docker required.

A common misconception: Docker isn't *part of* Fabric, it's just how the
official tutorial (`test-network`) happens to launch things for
convenience. The actual Fabric components are plain Go binaries:

- `orderer` — the ordering service, a normal executable
- `peer` — runs the ledger + validates transactions, a normal executable
- `cryptogen` / `configtxgen` — generate certs and channel configuration, normal executables
- chaincode — since Fabric 2.4, can run as **Chaincode-as-a-Service (CCaaS)**: a plain long-lived process the peer dials into over gRPC, instead of a Docker container the peer builds and launches itself

`blockchain/network/` in this project runs **all of the above as ordinary
background OS processes** on your Mac — no `docker`, no `docker-compose`,
nothing. This is a genuine, minimal, real Fabric network: one orderer, one
peer, one channel, real MSP identities generated by `cryptogen`, real
TLS, real signed/endorsed transactions, and your actual chaincode
(`blockchain/chaincode/lib/traceability.js`) executing exactly as it would
on a full multi-org production network.

The trade-off: this is a single-org network to keep the Docker-free setup
tractable. A real multi-org consortium (farmer cooperatives / testing labs
/ AYUSH regulator each as separate orgs) would still not need Docker — you'd
just repeat the `crypto-config.yaml` → `configtx.yaml` → peer-process
pattern once per org. That's noted inline in `crypto-config.yaml`.

## Step-by-step: install prerequisites (macOS + Homebrew, no Docker)

```bash
# Go — Fabric's binaries and cryptogen/configtxgen are Go programs
brew install go

# Node.js 18+ (you already have this for the frontend)
node -v

# git, curl, jq, python3 — usually already on macOS
brew install git curl jq
```

### Get the Fabric binaries only (skip Docker images entirely)

Fabric's official installer script lets you choose exactly what to fetch.
Pass `binary samples` — **not** `docker` — so nothing Docker-related is
ever pulled:

```bash
mkdir -p ~/fabric && cd ~/fabric
curl -sSLO https://raw.githubusercontent.com/hyperledger/fabric/main/scripts/install-fabric.sh
chmod +x install-fabric.sh
./install-fabric.sh binary samples
```

This gives you:
- `~/fabric/bin/` — `orderer`, `peer`, `cryptogen`, `configtxgen`, `osnadmin`, etc.
- `~/fabric/config/` — sample `core.yaml` / `orderer.yaml` templates
- `~/fabric/fabric-samples/` — includes `ccaas_builder/`, the small script bundle that teaches the peer how to talk to Chaincode-as-a-Service (still no Docker — these are shell/Go scripts, not containers)

Add the binaries to your PATH (put this in `~/.zshrc` too):
```bash
export PATH=~/fabric/bin:$PATH
```

## Step-by-step: bring up the network

All commands below assume you're inside `blockchain/network/` in this project.

```bash
cd /absolute/path/to/ayurveda-app-combined/blockchain/network
```

### 1. Generate identities and channel genesis block
```bash
./scripts/01-generate-crypto.sh
./scripts/02-generate-channel-block.sh
```

### 2. Prepare peer/orderer config for CCaaS (one-time)
```bash
./scripts/00-patch-core-yaml.sh ~/fabric/fabric-samples
```
This copies `core.yaml`/`orderer.yaml` from your Fabric install into
`blockchain/network/config/` and patches `core.yaml` so the peer knows to
reach chaincode via the `ccaas_builder` (a TCP dial, not a container).

### 3. Start the orderer and the peer (each in its own terminal tab, or backgrounded)
```bash
./scripts/03-start-orderer.sh
./scripts/04-start-peer.sh
```
Check `logs/orderer.log` and `logs/peer0.log` if either fails to come up —
almost always a path typo in the exported env vars.

### 4. Join the channel
```bash
./scripts/05-create-and-join-channel.sh
```

### 5. Start the chaincode itself (as a plain Node process — this is the "no Docker for chaincode either" part)

In a **separate terminal**:
```bash
cd ../chaincode
npm install
```
Leave this terminal open — you'll start the actual server in step 6 once
you have the package ID.

### 6. Deploy the chaincode definition
```bash
cd ../network
./scripts/06-deploy-chaincode-ccaas.sh
```
This script will pause and print a `CHAINCODE_ID` — switch to the chaincode
terminal from step 5 and run exactly what it tells you, e.g.:
```bash
CHAINCODE_ID=traceability_1.0:abcd1234... npm run start:server
```
Once you see `✓ traceability chaincode server listening on 127.0.0.1:9999`,
go back to the network terminal and press Enter to let the script finish
approving + committing the chaincode definition.

### 7. Sanity-check with the peer CLI
```bash
export FABRIC_CFG_PATH=$PWD/config
export CORE_PEER_LOCALMSPID=Org1MSP
export CORE_PEER_MSPCONFIGPATH=$PWD/crypto-config/peerOrganizations/org1.ayurveda.com/users/Admin@org1.ayurveda.com/msp
export CORE_PEER_ADDRESS=127.0.0.1:7051
export CORE_PEER_TLS_ENABLED=true
export CORE_PEER_TLS_ROOTCERT_FILE=$PWD/crypto-config/peerOrganizations/org1.ayurveda.com/peers/peer0.org1.ayurveda.com/tls/ca.crt
ORDERER_TLS_CA=$PWD/crypto-config/ordererOrganizations/ayurveda.com/orderers/orderer.ayurveda.com/tls/ca.crt

peer chaincode invoke -o 127.0.0.1:7050 --tls --cafile $ORDERER_TLS_CA \
  -C ayurveda-channel -n traceability \
  -c '{"function":"registerConservationZone","Args":["zone-1","Ashwagandha","Nagaur-Mandsaur Belt","25.15","74.85","120","11","3","50000"]}'

peer chaincode invoke -o 127.0.0.1:7050 --tls --cafile $ORDERER_TLS_CA \
  -C ayurveda-channel -n traceability \
  -c '{"function":"recordCollectionEvent","Args":["BATCH-001","Ashwagandha","farmer-1","25.15","74.85","2026-01-15","12.5","9.2"]}'

peer chaincode query -C ayurveda-channel -n traceability \
  -c '{"function":"getProvenance","Args":["BATCH-001"]}'
```
If the last command prints back a JSON `collectionEvent`, your Docker-free
Fabric network is genuinely live and working.

## Step-by-step: connect the Flask backend

### 8. Start the gateway bridge (Node → Fabric gRPC)
```bash
cd ../gateway-api
npm install
cp .env.example .env      # defaults already match this Docker-free layout
npm start
```
Confirm: `curl http://localhost:4000/health`

### 9. Point Flask at it
```bash
cd ../../backend
pip install -r requirements.txt --break-system-packages
export FABRIC_GATEWAY_URL=http://localhost:4000
python3 app.py
```
Every batch registration, processing step, lab test, dispatch and delivery
confirmation now submits a real transaction to your running Fabric network,
in addition to the local hash-chain. Confirm with:
```bash
curl http://localhost:5000/api/blockchain/status
```
→ `"hyperledger_fabric": {"enabled": true, "reachable": true, ...}`

### 10. Shutting everything down
```bash
# Ctrl+C in the chaincode server terminal and the gateway-api terminal
cd blockchain/network
./scripts/99-stop-network.sh
```
To wipe all ledger state and start fresh next time: `rm -rf data crypto-config channel-artifacts config`.

## Troubleshooting

- **`cryptogen`/`configtxgen`/`orderer`/`peer`: command not found** → your
  `PATH` doesn't include `~/fabric/bin`. `echo $PATH` and re-check step "Get
  the Fabric binaries".
- **Peer fails to start, log mentions externalBuilders / ccaas** → step 2
  (`00-patch-core-yaml.sh`) either wasn't run or couldn't auto-patch —
  open `blockchain/network/config/core.yaml`, find `chaincode:`, and add
  the `externalBuilders` block by hand (the script prints the exact YAML
  to paste if it can't do it automatically).
- **`osnadmin channel join` connection refused** → the orderer isn't up yet,
  or `ORDERER_ADMIN_LISTENADDRESS` in `03-start-orderer.sh` doesn't match
  the `-o` flag in `05-create-and-join-channel.sh` (both should be `:7053`).
- **Chaincode invoke times out / "failed to connect to 127.0.0.1:9999"** →
  the chaincode server (step 5/6, `server.js`) isn't running, or was
  started with the wrong `CHAINCODE_ID`. Restart it with the exact ID
  from `peer lifecycle chaincode queryinstalled`.
- **Ports already in use (7050/7051/7053/9999/4000)** → something from a
  previous run is still alive. `./scripts/99-stop-network.sh`, then
  `lsof -i :7050` (etc.) to find and kill stragglers.

## Why a Node bridge (`gateway-api`) between Flask and Fabric?

Fabric's officially-supported, actively-maintained client SDKs are
Node.js, Go and Java — there's no first-party Python SDK. Rather than lean
on an unofficial/stale community Python binding for something this central,
`gateway-api/` is a thin Node/Express service using the real
`@hyperledger/fabric-gateway` SDK, exposing plain REST that Flask calls.
This is the same pattern most production systems use when their main
application isn't in one of Fabric's native-SDK languages, and it means
this chaincode/network setup could back a Java or Go application later with
zero changes to the network or chaincode itself.

## Two ledger layers — what's what

1. **`/backend`'s hash-chain (`audit_log` table)** — always on, zero extra
   infrastructure, works today on Render/Neon even with no Fabric network
   running. Genuinely tamper-evident (SHA-256 chained rows), but a single
   database, not a distributed ledger.
2. **This Fabric network** — real, multi-process, cryptographically signed
   and endorsed. When `FABRIC_GATEWAY_URL` is set, every write goes to
   *both* layers; unset it and the app runs exactly as before.
